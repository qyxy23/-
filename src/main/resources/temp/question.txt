# 海龟汤项目 - Java后端开发技术要点总结

## 项目概述
海龟汤（HaiGuiTang）是一款基于Web的社交推理游戏平台，支持AI智能对话、群组聊天、好友互动等功能。项目采用前后端分离架构，提供完整的用户管理和即时通讯能力。

---

## 一、后端技术栈（Spring Boot 3.3.4）

### 1. 核心框架
- **Spring Boot 3.3.4** - 基础框架
- **Spring Security** - 安全认证与授权
- **Spring Data JPA** - 数据持久化ORM框架
- **Spring WebSocket** - WebSocket实时通信支持

### 2. 数据库技术
- **MySQL 8.0.33** - 关系型数据库
- **Druid 1.2.20** - 阿里巴巴数据库连接池
- **MyBatis 3.0.3** - 持久层框架（与JPA混合使用）
- **PageHelper 2.1.0** - 分页插件
- **Redis (Redisson 3.41.0)** - 缓存与会话管理
  - Redis Stack支持（Letture 6.3.2 + Lettucemod 4.5.0）
  - 支持向量搜索功能（RediSearch）

### 3. 安全与认证
- **Spring Security** - 认证授权框架
  - JWT（jjwt 0.11.5）- 无状态Token认证
  - BCrypt密码加密
  - 基于角色的访问控制（RBAC）
  - 自定义JWT过滤器（JwtAuthenticationFilter）
- **CORS跨域配置** - 支持多域名访问

### 4. 消息队列
- **Spring AMQP (RabbitMQ)** - 异步消息处理
  - 手动确认模式
  - 消费者并发控制（5-10并发）
  - 重试机制（最多3次）
- **RocketMQ 2.2.3** - 备选消息队列（已集成）

### 5. 实时通信
- **WebSocket (Stomp)** - 即时消息推送
  - 私聊功能（PrivateChatService）
  - 群聊功能（GroupService）
  - 游戏房间聊天（RoomService）
  - WebSocket安全配置

### 6. 文件存储
- **MinIO 8.5.7** - 对象存储服务
  - 用户头像存储
  - 文件上传支持（最大10MB）

### 7. AI集成
- **火山引擎 Ark Runtime 0.2.34** - AI模型调用
- **向量数据库（Redis Stack）** - AI上下文向量存储
- **BGE向量服务** - 文本向量化

- **Hutool 5.7.17** - Java工具类库

### 9. API文档

- **Controller层** - 11个控制器，处理HTTP请求

  - ChatWithAiController（AI聊天）
  - ChatWithFriendController（好友聊天）
  - ChatGroupController（群组聊天）
  - ChatRoomController（游戏房间）
  - FriendsController（好友管理）
  - HaiGuiTangController（海龟汤游戏）

- **Service层** - 业务逻辑处理

- **Mapper层** - 数据访问（MyBatis）

## 五、技术难点与解决方案

### 1. 高并发实时通讯
**挑战**: 多用户同时在线，消息实时推送
**解决方案**:
- WebSocket长连接 + STOMP协议
- Redis存储在线用户状态
- 自定义SessionRegistry管理会话
- RabbitMQ异步消息处理

### 2. 消息已读状态同步
**挑战**: 群组消息多人已读状态管理
**解决方案**:
- 独立的已读状态表（GroupMessageRead、ChatGameMessageRead）
- 复合主键设计（GroupId/ChatGameId + UserId + MessageId）
- 异步更新已读状态，减少主流程压力

### 3. 分布式会话管理
**挑战**: 多服务器部署时用户会话一致性
**解决方案**:
- JWT无状态认证
- Redis存储用户在线状态和Token
- Redisson分布式锁防止并发问题

### 4. AI上下文管理
**挑战**: AI对话需要长期上下文记忆
**解决方案**:
- 向量数据库（Redis Stack RediSearch）
- BGE模型文本向量化
- 向量元数据存储（VectorMetadata）

### 5. 数据库混合使用
**挑战**: MyBatis和JPA混合使用的事务管理
**解决方案**:
- @Transactional统一事务管理
- 复杂查询用MyBatis，简单CRUD用JPA
- 统一的Repository/Mapper分层

### 6. 跨平台兼容性
**挑战**: 前端需要支持H5、小程序、APP多端
**解决方案**:
- UniApp跨平台框架
- 条件编译处理平台差异
- 统一的后端API接口

## 六、项目架构说明

### 架构类型
**单体应用（Monolithic Application）** - 不是微服务架构

### 架构特点
1. **单体应用的优势**
   - 开发简单：一个项目，一个IDEA工作空间
   - 部署方便：打包成一个jar文件，直接运行
   - 调试容易：本地调试，无需多个服务
   - 事务管理简单：本地事务即可，不需要分布式事务

2. **前后端分离**
   - 后端：Spring Boot单体应用
   - 前端：UniApp + Vue3（支持H5、小程序、APP）
   - 通信：RESTful API + WebSocket

3. **使用分布式技术增强单体应用**
   - **Redis**: 缓存、会话管理、在线状态（不是微服务间通信）
   - **RabbitMQ**: 异步任务、系统解耦（不是服务间消息）
   - **Redisson**: 分布式锁（防止多实例部署时的并发问题）
   - **MinIO**: 分布式文件存储

4. **支持水平扩展**
   - 通过部署多个应用实例 + 负载均衡
   - 使用Redis共享会话状态
   - 使用Redisson分布式锁解决并发问题

### 与微服务架构的区别
| 特性 | 您的项目（单体应用） | 微服务架构 |
|------|---------------------|------------|
| 服务数量 | 1个应用 | 多个独立服务 |
| 服务通信 | 无需服务间通信 | 服务间需要RPC/HTTP调用 |
| 数据库 | 共享一个数据库 | 每个服务独立数据库 |
| 部署 | 打包成一个jar | 每个服务独立部署 |
| 复杂度 | 较低 | 较高（需要服务发现、网关等） |
| 开发效率 | 高（适合中小型项目） | 低（适合大型复杂项目） |

### 为什么选择单体应用？
1. **项目规模适中** - 用户量和功能模块不需要微服务
2. **团队规模** - 小团队开发，单体应用更容易维护
3. **降低复杂度** - 避免微服务带来的分布式事务、服务治理等复杂问题
4. **快速迭代** - 单体应用更容易快速开发和部署新功能

### 未来可以升级为微服务
当项目发展到以下情况时，可以考虑微服务拆分：
- 团队规模扩大（20+人）
- 单体应用代码量过大（几十万行）
- 部署和发布时间过长
- 不同模块需要独立扩容

## 八、面试可能被问到的问题

### 1. 技术问题
- Q: 为什么同时使用MyBatis和JPA？
  A: 复杂查询（多表关联、动态SQL）用MyBatis灵活控制，简单CRUD用JPA提高开发效率

- Q: JWT如何保证安全？
  A: 使用BCrypt加密密码，Token签名使用密钥，设置过期时间，Redis存储黑名单

- Q: 如何处理WebSocket连接管理？
  A: 自定义SessionRegistry，Redis存储在线用户，心跳检测，断线重连机制

- Q: 消息已读状态如何设计？
  A: 独立的已读状态表，复合主键（群组ID + 用户ID + 消息ID），异步更新

- Q: 如何保证AI上下文连续性？
  A: 向量数据库存储历史对话向量，BGE模型向量化，语义检索相关上下文

- Q: RabbitMQ如何保证消息不丢失？
  A: 手动确认模式，持久化队列，死信队列，重试机制

- Q: 如何优化分页查询？
  A: PageHelper物理分页，索引优化，避免select *，使用缓存

### 2. 架构问题
- Q: 如何支持分布式部署？
  A: JWT无状态认证，Redis共享会话，RabbitMQ异步通信，分布式锁

- Q: 系统如何做监控？
  A: [根据实际情况回答，如日志记录、性能监控等]

### 3. 业务问题
- Q: 游戏房间如何防止人数超限？
  A: Redisson分布式锁，原子操作检查人数

- Q: 如何处理AI推理超时？
  A: 异步任务+超时控制，熔断降级

public class TokenInterceptor implements HandlerInterceptor {
    @Resource
    private JwtTokenUtil jwtTokenUtil;

    public boolean preHandle(HttpServletRequest request, HttpServletResponse response, Object handler) {
        // 1. 从请求头获取Token
        String token = request.getHeader("Authorization");

        // 2. 验证Token

        }

## 设计模式使用建议

### 1. 策略模式 - 最佳实践
✅ **使用场景**: 需要在运行时动态选择算法
✅ **项目应用**: 登录/注册、支付方式、消息推送策略
❌ **避免**: 如果只有一种实现，不要过度使用策略模式

### 2. 仓储模式 - 最佳实践
✅ **使用场景**: 数据访问层，封装数据库操作
✅ **项目应用**: 所有Repository接口
❌ **避免**: 不要在Repository中编写业务逻辑

### 3. 服务层模式 - 最佳实践
✅ **使用场景**: 业务逻辑处理，协调多个Repository
✅ **项目应用**: 所有Service接口和实现类
❌ **避免**: Service层不要直接访问HTTP请求参数（由Controller处理）

### 4. 拦截器模式 - 最佳实践
✅ **使用场景**: 身份认证、权限验证、日志记录
✅ **项目应用**: Token验证、WebSocket安全拦截
❌ **避免**: 不要在拦截器中修改业务数据

### 5. 外观模式 - 最佳实践
✅ **使用场景**: 简化复杂子系统的调用
✅ **项目应用**: AIManager封装AI服务调用
❌ **避免**: 不要限制子系统提供的灵活性

### 6. 观察者模式 - 最佳实践
✅ **使用场景**: 事件驱动、异步处理、解耦
✅ **项目应用**: WebSocket事件监听、消息队列消费
❌ **避免**: 注意事件处理的顺序和性能问题

### 7. 构建者模式 - 最佳实践
✅ **使用场景**: 复杂对象创建、可选参数多
✅ **项目应用**: ChatMessage、ChatCompletionRequest等
❌ **避免**: 简单对象不要使用构建者模式（直接new更高效）

### 8. DTO模式 - 最佳实践
✅ **使用场景**: 层与层之间数据传输、前后端交互
✅ **项目应用**: 所有Request/Response对象
❌ **避免**: 不要创建过多的中间DTO（不必要的转换开销）
